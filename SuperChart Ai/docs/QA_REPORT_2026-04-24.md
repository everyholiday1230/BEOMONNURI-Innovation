## 🔴 Blocker 급 (서비스 오픈 전 반드시 해결)

### 1. [프론트] 회원가입 시 "이용약관/개인정보처리방침" 링크 클릭 → `ReferenceError: _showTerms is not defined`
- **위치(수정 전)**: `static/index.html` 2797~2825 라인 — `btn.onclick` 핸들러의 `try { ... } else { ... }` 분기 내부에 `window._showTerms` 정의가 들어가 있었음.
- **증상**: 사용자가 처음 회원가입 모달을 열고 약관/개인정보 링크를 클릭하면 함수가 아직 정의되지 않아 오류. 체크박스 동의 없이는 가입 불가 → **사실상 신규 가입 차단**.
- **수정**: `_showTerms`를 최상위(showAuth 바로 위)로 옮겨 페이지 로드 시점에 무조건 전역에 등록되도록 변경.
- **상태**: ✅ 수정 완료

### 2. [DB] DDL에 테이블 5종 누락 → 운영 시 즉시 500 에러
- `src/models/tables.py`는 정의되어 있으나 `src/db/ddl.sql`에 생성문이 없는 테이블:
  - `access_logs` (로그인 시도/감사 로그 기록)
  - `notices` (공지 배너/목록)
  - `faqs`
  - `site_settings` (점검모드 플래그!)
  - `user_chart_settings`
- **증상**:
  - 로그인 → `db.add(AccessLog(...))` → `relation "access_logs" does not exist` 500
  - 메인 진입 시 점검모드 체크 → `SiteSetting` 조회 실패 → 미들웨어에서 조용히 예외 먹고 통과(다행), 하지만 공지 배너/FAQ/차트 설정 저장은 전부 실패
- **수정**: `src/db/ddl.sql` 말미에 5개 테이블 + 인덱스 + `site_settings` 기본값 시드를 `CREATE TABLE IF NOT EXISTS`로 추가.
- **상태**: ✅ 수정 완료

### 3. [보안 심각] 비밀번호 재설정 API가 무인증 → 누구나 타인 계정 비밀번호 초기화 가능
- **위치(수정 전)**: `POST /v1/auth/reset-password` — 인증 없이 `email`만 받으면 임시 비밀번호를 발급하고 응답에 그대로 노출.
- **공격 시나리오**: 공격자가 피해자의 이메일만 알면 임시 비밀번호를 취득 → 즉시 계정 탈취. 추가로 프론트 UI(`_doResetPw`)가 발급된 임시 비밀번호를 화면에 대놓고 출력.
- **수정**:
  - 서버: `_auth_admin_check` 적용 (ADMIN_KEY 필요)
  - 프론트: 자가 재설정 UI를 "점검 중" 메시지로 대체 (이메일 인증 시스템 도입 전까지 비활성)
- **상태**: ✅ 수정 완료

### 4. [보안 심각] 실전매매(`/v1/trading/live/*`) 엔드포인트 전부 무인증
- **위치(수정 전)**: `src/api/trading.py`
  - `_require_admin` 함수는 **정의만 되어 있고 어디에도 적용되지 않음**
  - `/live/start`, `/live/stop`, `/live/status`, `/live/set-amount`, `/live/set-symbols` 전부 누구나 호출 가능
  - 또한 `all_markers` 안에서 `live_trader_v2`를 **import 전에** 사용 → NameError 런타임 크래시
- **공격 시나리오**: 외부에서 `POST /v1/trading/live/set-amount {"amount": 999999, "leverage": 125}` → 사용자 자금으로 고배율 거래 실행.
- **수정**:
  - `live_trader_v2` import를 파일 최상단으로 이동
  - 모든 `/live/*` 엔드포인트에 `_require_admin` 강제 적용
  - `set-amount`에 amount/leverage 범위 검증, `set-symbols`에 심볼 화이트리스트 정규식 검증 추가
- **상태**: ✅ 수정 완료

### 5. [보안] 기본 JWT_SECRET이 그대로면 서버가 기동됨
- **위치(수정 전)**: `src/config.py` — 단순 `jwt_secret: str` 만 선언. `.env.example`/`.env.docker.example`/`.env.docker`에 `change-me-in-production` 기본값.
- **위험**: 운영 환경에서 실수로 그대로 배포 시, 공격자가 JWT를 위조하여 임의 사용자로 로그인 가능.
- **수정**:
  - `src/config.py`에서 기동 시 JWT_SECRET 길이(≥32자) + 약한 값 블랙리스트 체크
  - `ENV=prod|production|live` 면 약한 값 감지 시 즉시 `RuntimeError`로 기동 거부
  - 그 외 환경은 경고만 출력
  - `.env.example` / `.env.docker.example` 문구를 `CHANGE_ME_TO_A_LONG_RANDOM_STRING_...`로 변경 + 생성 커맨드 안내
- **상태**: ✅ 수정 완료

### 6. [보안] `ADMIN_KEY`가 빈 문자열일 때 `k == ""` 로 관리자 API 통과 가능
- **위치(수정 전)**: `src/api/site.py::_check_admin` 및 `src/api/auth.py` 관리자 핸들러들
  - `if not k or k != ADMIN_KEY():` 패턴에서 환경변수가 비어 있고 요청 헤더/쿼리도 비어 있으면 `k = ""`, `ADMIN_KEY() = ""` → `"" != ""` 은 False → `not k`가 True 라 다행히 막히긴 하지만, **명시적으로 "관리자 키 미설정"을 실패로 처리하는 게 안전**
- **수정**: `admin = ADMIN_KEY(); if not admin: raise 403` 을 먼저 검사해 미설정 = 관리자 API 전면 비활성화.
- **상태**: ✅ 수정 완료

### 7. [프론트 XSS] 뉴스 피드 제목/설명이 escape 없이 innerHTML에 삽입
- **위치(수정 전)**: `static/index.html` 2573~2583 — `n.title`, `n.description`, `n.source`, `n.url` 을 innerHTML에 그대로 템플릿 리터럴로 삽입.
- **공격 시나리오**: 외부 뉴스 피드(CoinDesk/CoinGecko 등) 중 하나가 오염(또는 공격자 소유 피드가 `news.py`에 추가)되면 `<img onerror=...>` 류 페이로드로 세션 탈취 가능. 특히 `<a href="${n.url}">`에 `javascript:` 스킴이 들어가면 클릭 시 실행.
- **수정**:
  - 전역 `_escHtml()` / `_safeUrl()` 헬퍼 추가 (javascript/data/vbscript 스킴 차단)
  - 뉴스 렌더링의 `title / description / source / summary / timeStr / url` 전부 escape 적용
- **상태**: ✅ 수정 완료
